Task 5 — Compile / run CLI (toolchain boundary)

  Task 5 is the user-facing toolchain step: split compiling Sanskrit source from running bytecode, so programs are
  shipped and executed as portable .sskbc artifacts instead of always going through the Python parser at run time.

  Where things stand today

  .ssk  →  parse → IR → bytecode  →  VM  →  output
           └──────── all in one `run` ────────┘

  • sanskript run foo.ssk (or sanskript foo.ssk) calls interpreter.run(), which parses, compiles, and executes in a
    single pass.
  • The bytecode layer already exists: encode_program, decode_program, validate_bytecode, dump_bytecode_file,
    load_bytecode_file in bytecode.py.
  • Conformance JSON fixtures and ssk-vm can execute bytecode without re-parsing Sanskrit.
  • What’s missing is a stable CLI workflow and conventions around .sskbc files.

  What Task 5 should deliver

  ┌───────────────────────────┬────────────────────────────────────────────────────────────────────────────────────────┐
  │ Piece                     │ Purpose                                                                                │
  ├───────────────────────────┼────────────────────────────────────────────────────────────────────────────────────────┤
  │ sanskript compile         │ Read .ssk → parse → compile → write .sskbc (JSON matching v1/v2 schema)                │
  │ sanskript run (extended)  │ Accept either .ssk (compile+run, today’s behavior) or .sskbc (load + VM only)          │
  │ File convention           │ e.g. program.ssk → program.sskbc; document version field and optional metadata         │
  │ Validation at compile     │ Run validate_bytecode before writing; fail fast with clear errors                      │
  │ time                      │                                                                                        │
  │ Tests                     │ Round-trip: compile example → load → same output as compile_source; CLI integration    │
  │                           │ tests                                                                                  │
  │ Docs                      │ Short “toolchain” section: when to compile, format of .sskbc, how it relates to ssk-vm │
  └───────────────────────────┴────────────────────────────────────────────────────────────────────────────────────────┘

  Optional but natural follow-ons (can be in or just after Task 5):

  • --output / -o for compile path
  • --vm python|rust to delegate execution to ssk-vm when installed
  • sanskript disasm (pretty-print bytecode) for debugging
  • Cache/stale detection (recompile if .ssk newer than .sskbc)

  Why it matters in the roadmap

  ┌──────┬─────────────────────────────────────────────────────────────────────────┐
  │ Task │ Role                                                                    │
  ├──────┼─────────────────────────────────────────────────────────────────────────┤
  │ 1    │ Wire format (bytecode spec)                                             │
  │ 2–3  │ What the compiler can emit (control flow, procedures)                   │
  │ 4    │ Second implementation of the executor (Rust VM)                         │
  │ 5    │ Second implementation of the delivery path (compile once, run anywhere) │
  └──────┴─────────────────────────────────────────────────────────────────────────┘

  Task 5 is what makes Tasks 1 and 4 useful to humans: you can hand someone a .sskbc and run it with Python or Rust
  without dragging the whole morphology/parser stack into the hot path (even if you still use Python for compilation for
   now).

  What Task 5 is not

  • Not rewriting the compiler in Sanskript (that’s later self-hosting).
  • Not expanding the language (no new opcodes/frames—that was 2–3).
  • Not requiring Rust for compile; compile stays in Python initially; Rust is for run via ssk-vm.
  • Not a package manager, linker, or multi-file build graph (unless you explicitly scope that in).

  Rough acceptance bar (“done”)

  1. sanskript compile examples/caturtha.ssk writes a valid .sskbc that passes schema validation.
  2. sanskript run examples/caturtha.sskbc prints the same lines as sanskript run examples/caturtha.ssk.
  3. All new examples (shashṭha-if.ssk, etc.) compile and run from bytecode.
  4. Tests in CI; README updated with the two-step workflow.

  Dependencies / risks

  • Depends on: stable bytecode encode/decode (done), VM behavior (done), parser not eating trailing statements (antam
    etc.—done).
  • Risk: compile time still loads the large controlled lexicon (slow but acceptable for v1 of the CLI).
  • Risk: deciding whether .sskbc is plain JSON or a wrapped format (magic bytes, compression)—start with JSON for
    transparency and conformance parity.

  ────────────────────────────────────────

  In one line: Task 5 is the compile/run command-line split so Sanskript programs become portable bytecode files that
  any conforming VM (Python today, Rust via ssk-vm) can execute without recompiling from source.
