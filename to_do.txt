Here is everything from that discussion that is realistically doable in a few months (roughly 8–12 weeks), assuming
  focused part-time-to-full-time effort. Items are grouped by track; none of them require solving “full automatic
  Aṣṭādhyāyī for arbitrary Sanskrit.”

  ────────────────────────────────────────

  Morphology / derivation backend

  [x] 1. DerivationIntent API — typed request object (tinanta, subanta, kṛt, taddhita, samāsa, sandhi) with lemma, lakāra,
     case, person, etc.

  [x] 2. RecipePlanner — maps each intent kind to an ordered sūtra/engine recipe (not global search across all 3983
     sūtras).

  [x] 3. MorphologySynthesizer / synthesize() — intent → recipe → PaninianDerivationEngine (derive_sequence
     and/or family-scoped derive) → Analysis.

  [x] 4. Directed tiṅ synthesis — dhātu + lakāra + person/number/pada → finite verb surface (controlled dhātupatha).

  [x] 5. Directed sup synthesis — prātipadika + case + number → nominal surface (controlled stems).

  [x] 6. Directed kṛt synthesis — dhātu/stem + kṛt suffix → participle/gerund/etc. for a small inventory.

  [x] 7. Directed taddhita synthesis — stem + semantic relation + suffix → derived nominal (extend existing 4–5 engines).

  [x] 8. Directed samāsa synthesis — member analyses + compound type → surface (existing SamasaDerivationEngine).

  [x] 9. Directed sandhi join — left + right → joined form for controlled padaccheda rules (existing
     SandhiDerivationEngine).

  [x] 10. Family-scoped auto-derive — auto_derive() delegates to synthesize(); prefix-scoped derive for kṛt/taddhita
      when explicit sūtra sequences do not surface a form.

  [x] 11. Recipe catalog per grammar-register entry — document + code: “this lexeme is produced by this recipe.”

  [x] 12. Unify duplicate morphology — tiṅ/sup register synthesis routes through PaninianDerivationEngine +
      TinantaLakaraEngine/SubantaSupEngine; paradigm tables remain internal engine fallbacks.

  ────────────────────────────────────────

  Lexicon & build pipeline (highest ROI)

  [x] 13. build_controlled_lexicon.py — offline script that calls synthesize() for every allowed form in the grammar
      register.

  [x] 14. data/controlled_lexicon.json — versioned artifact checked into repo or built in CI.

  [x] 15. Load lexicon from artifact — morphology.build_lexicon() loads JSON and merges hand-maintained overrides
      from data/lexicon_overrides.json.

  [x] 16. Lexicon diff / validation test — CI fails if register changes but lexicon wasn’t regenerated.

  [x] 17. Authoring workflow — add a new verb/noun to register → run builder → lexicon updates automatically.

  ────────────────────────────────────────

  Language hot path (parser / morphology)

  [x] 18. MorphologyFacade — single entry: analyze_token(), analyze_sentence().

  [x] 19. Cache-first lookup — LEXICON / JSON first; no engine call on hot path for known forms.

  [x] 20. Swap parser to facade — one-line architectural change in parser.py import path.

  [x] 21. In-memory parse cache — memoize unknown-token resolution within a session.

  [x] 22. Strict register enforcement — clear error: “form not in controlled register” (better than silent lexicon miss).

  [x] 23. Kāraka validation pass — role↔case consistency for all participants; KarakaVibhaktiEngine cross-check
      for frame-required roles (with nominative kartṛ register convention).

  [ ] 24. Frame registry cleanup — verb frames driven from data (JSON/YAML) instead of hard-coded if verb.surface == ...
      branches where possible.

  [ ] 25. Normalize input sandhi — optional pre-tokenization step using existing sandhi join/split for controlled
      examples.

  [x] 26. Devanagari → IAST normalize — extend morphology.normalize() on the hot path (already planned in README).

  ────────────────────────────────────────

  Parser / compiler / runtime (language features on top)

  [ ] 27. Expand controlled verb frames — more verbs wired through same Assign/Increase/Display pattern.

  [ ] 28. Expand controlled noun/pronoun/numeral inventory — via lexicon builder, not hand LEXICON entries.

  [ ] 29. Data-driven parser dispatch — table from (verb_lemma, frame) → AST constructor instead of growing if chains.

  [ ] 30. Richer Analysis on hot path — propagate lakāra, pada, gender from synthesized lexicon into parser decisions.

  [ ] 31. IR/compiler: new statement forms — conditional, loop, function call (language design already sketched in docs).

  [ ] 32. VM opcodes for new semantics — extend bytecode + VM for anything new in IR (still integer-only is fine for v1).

  [ ] 33. Better runtime errors — morphological, parse, and VM errors with Sanskrit-aware messages.

  [ ] 34. Stdlib in Sanskrit surface forms — generated via synthesizer, loaded as prebuilt lexicon entries.

  [ ] 35. Example programs beyond prathama.ssk — exercise new frames and lexicon entries.

  ────────────────────────────────────────

  Engine / foundation hardening (bounded, not new sūtra mining)

  [ ] 36. Freeze sūtra layer — bugfixes only; document “engine API stable.”

  [x] 37. Document integration boundary — derive_paninian() / synthesize() / auto_derive() as the public morphology
      backend API (exported from sanskript package).

  [x] 38. Integration tests: register ↔ lexicon ↔ parse — end-to-end: synthesize form → in lexicon → parses → compiles →
       runs.

  [x] 39. Per-family synthesis tests — tiṅ, sup, kṛt, taddhita, samāsa, sandhi golden tests in test_morphology_synth.py.

  [ ] 40. Performance baseline — lexicon lookup + full parse/compile of examples under acceptable ms (sanity, not
      optimization project).

  [ ] 41. Regenerate / shrink predicate-only-weak-sutras.md — optional audit script in CI (maintenance, not new grammar
      work).

  ────────────────────────────────────────

  Reverse analysis (limited, still feasible in months)

  [x] 42. Generate-and-test analyzer for controlled register only — unknown token → try candidates from register
      inventory × recipes → match surface (MorphologyFacade._analyze_unknown).

  [x] 43. Ambiguity reporting — if multiple analyses match, raise structured error (don’t silently pick).

  [x] 44. Timeout / bounded search — cap candidates per token so parse stays predictable.

  Not feasible in months: unique analysis for arbitrary Classical Sanskrit surfaces.

  ────────────────────────────────────────

  Docs & tooling

  [ ] 45. Update README / implementation-plan — “morphology compiler backend + lexicon builder” as official architecture.

  [~] 46. Grammar register ↔ code sync — single source listing allowed lemmas and derivations.
      (grammar_register.py is source of truth; docs/grammar-register.md not yet synced)

  [x] 47. CLI: sanskript build-lexicon and sanskript synthesize for debugging derivations.

  [x] 48. CLI: sanskript analyze <token> — show Analysis + derivation trace for developers.

  ────────────────────────────────────────

  Explicitly NOT in the “few months” list

  These were discussed but should not be on a near-term plan:

  • Full automatic rule discovery across all 3983 sūtras for any input
  • Complete tripādī / asiddha closure for open-ended derivation
  • Open-vocabulary live engine derivation on every parse token at scale
  • Full compound segmentation + sandhi tokenizer for free Sanskrit text
  • Replacing the entire hand-tuned tinanta/subanta modules in one pass
